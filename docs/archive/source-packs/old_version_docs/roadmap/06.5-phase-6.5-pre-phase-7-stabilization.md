# 06.5 — Phase 6.5: Pre-Phase-7 Stabilization and Surface Cleanup

This document remains historical. If any non-UI backend/runtime cleanup listed here is still unfinished in the current codebase, it is now carried forward into **Phase 13** rather than treated as live Phase 6.5 work.

## Goal

Tighten control truth, simplify transition ownership, and clean the operator/docs surface before controller-driven looping lands.

This phase is **not** a new runtime model.
It is a short stabilization and simplification pass between validated phase-6 flow-first execution and phase-7 controller/governance work.

## Why this phase exists

The flow-first runtime reset landed, but several important preconditions for Phase 7 are still too loose:

- a few external write paths can still mutate stale or terminal runtime state
- transition ownership is spread across multiple runtime modules
- retry / watchdog / replan / resume semantics are not yet normalized
- operator/API surface still exposes too many controller/debug internals
- front-door repo/docs pages still tell different stories about current project state

Phase 7 should not formalize liveness and governance on top of those gaps.

## Keep this phase narrow

Phase 6.5 should:

- harden control integrity
- reduce duplicated transition logic
- clarify release/resume boundaries
- clean the public/operator surface
- fix the repo's current-state story

Phase 6.5 should **not** become:

- a new state model
- a broad product redesign
- a full event-stream project
- a full policy engine rollout
- a large console rewrite

## P0 — Must fix before Phase 7 starts

### 1. Control integrity: reject stale or terminal control writes

**Problem**

A few current write paths are too permissive. Stale or terminal attempts can still rewrite runtime truth.

**Hot files**

- `apps/api/app/runtime/checkpoints.py`
- `apps/api/app/runtime/approvals.py`
- `apps/api/app/runtime/dispatcher.py`
- `apps/api/app/runtime/watchdog.py`
- `apps/api/app/runtime/runner.py`
- `apps/api/app/runtime/replan.py`

**Required cleanup**

Add one shared internal guard for external/runtime control inputs:

- manifest acknowledgement
- checkpoint write
- approval resolution
- operator retry
- watchdog recovery
- replan adoption/requester validation

That guard should prove:

- target attempt is the current/latest attempt for the node
- target attempt belongs to the active flow revision
- target attempt/session binding is still current
- the current attempt status is eligible for this write
- terminal/superseded/aborted/cancelled attempts cannot mutate control state

**Specific integrity gaps to close**

- late approval must not reopen a completed node
- checkpoint writes must not rewrite finished attempt history
- pause must be sticky; manifest acknowledgement must not punch through it
- approval resolution must not mark flow `running` unless the derived next boundary actually permits it
- watchdog must keep session truth aligned with attempt/node truth

### 2. Shared transition ownership

**Problem**

Transition logic is still split across:

- `runner.py`
- `checkpoints.py`
- `approvals.py`
- `dispatcher.py`
- `watchdog.py`
- `replan.py`

Each module mutates flow/node/attempt/session/approval/manifest state slightly differently.

**Required cleanup**

Introduce one small internal transition layer that owns common lifecycle operations such as:

- start/resume attempt
- block attempt
- finish attempt
- abort/cancel attempt
- end/reset node session
- expire pending approvals
- supersede projected manifests
- complete flow if all nodes are done

The goal is not abstraction for its own sake.
The goal is to stop encoding the same state-transition rules in several places.

### 3. Normalize retry / watchdog / replan / resume semantics

**Problem**

These boundary semantics are still inconsistent:

- checkpoint `retry` does not yet behave like operator retry
- watchdog block does not yet have one formal recovery rule
- `request_replan()` still behaves as propose + validate + compile + adopt in one call
- waiting/release reasons are not derived in one place

**Hot files**

- `apps/api/app/runtime/runner.py`
- `apps/api/app/runtime/checkpoints.py`
- `apps/api/app/runtime/watchdog.py`
- `apps/api/app/runtime/replan.py`
- `apps/api/app/runtime/scheduler.py`
- `../../../../flows/04-approval-and-watchdog.md`

**Required cleanup**

Freeze the minimum runtime contract before Phase 7:

- what checkpoint `retry` means
- when a fresh `node_attempt` is created
- whether watchdog resumes same attempt, forces retry, or escalates
- what preconditions are required before replan adoption
- what release reasons make a waiting node resumable

### 4. Surface discipline: separate operator actions from controller/debug internals

**Problem**

The runtime is flow-first, but the API/console surface still exposes too many low-level controller actions as normal operator features.

**Hot files**

- `apps/api/app/api/routes/tasks.py`
- `apps/api/app/api/routes/compiler.py`
- `apps/api/app/api/routes/flows.py`
- `apps/api/app/api/routes/approvals.py`
- `apps/api/app/runtime/read_models.py`
- `apps/api/app/api/presenters/runtime.py`
- `apps/console/src/App.tsx`
- `apps/console/src/lib/api.ts`

**Required cleanup**

Split surface vocabulary into:

- **operator actions**
  - start
  - resume / continue
  - pause
  - cancel
  - approve / reject / not-required
  - retry
  - request replan
- **internal/debug/admin actions**
  - raw checkpoint ingest
  - raw manifest acknowledgement
  - watchdog trigger
  - low-level compile/debug routes

Also tighten:

- retry visibility so `waiting` is not treated as generically retryable
- replan provenance so requester node/attempt is explicit
- operator summary so it is not a single oversized debug dump
- context/session exposure so default views are filtered and summarized

### 5. Repo truth and front-door docs

**Problem**

The deep architecture/roadmap docs are mostly coherent, but the repo entry points and indexes still drift.

**Hot files**

- `ROADMAP.md`
- `README.md`
- `examples/demo-data/README.md`

- `../../../../architecture/05-diagrams-and-mermaid.md`

**Required cleanup**

Before Phase 7 begins, the repo should tell one honest story:

- runtime reset has landed
- flow-first runtime is live
- current work is Phase 6.5 stabilization
- Phase 7 follows after Phase 6.5 exit criteria are met

Also:

- label flow docs clearly as baseline vs target vs next-stage
- trim stale diagrams that teach the wrong delegation/bootstrap boundary
- stop describing `suggestion.md` as a cutover plan if it now behaves as a style/verification guide

### 6. Invariant-focused tests

**Problem**

Happy-path runtime coverage is decent, but the edges that matter for Phase 7 are under-tested.

**Hot files**

- `apps/api/tests/integration/test_runtime_api.py`
- `apps/api/tests/integration/test_flow_runtime_db.py`
- `apps/api/tests/integration/test_phase456_runtime_db.py`
- runtime tests/helpers that duplicate continue → ack → checkpoint choreography

**Required cleanup**

Add or tighten tests for:

- no writes to terminal attempts
- approval-after-green is rejected
- multi-blocker status recomputation
- continue-after-watchdog semantics
- session-state consistency after watchdog/retry/cancel/replan
- retry/failure timestamps (`finished_at`) are set correctly

## P2 — Explicitly not required before Phase 7

These are real future tasks, but they should not block Phase 7 once Phase 6.5 exit criteria are met:

- full typed event-stream productization beyond the minimum needed for controller correctness/auditability
- broad policy engine rollout beyond the minimum runtime-enforced policy surface
- full operator console redesign
- pack UX/productization
- deep repo-wide folder reorganization that does not change Phase-7 safety/readability materially

## Exit criteria

Phase 6.5 is complete when all of the following are true:

- external/runtime writes cannot mutate stale or terminal attempts
- pause, approval, context-ack, watchdog, retry, and replan boundaries have explicit current-attempt guards
- shared transition helpers own the core lifecycle mutations
- retry/watchdog/replan/resume semantics are frozen and documented in one place
- operator/public surfaces are clearly separated from internal/debug/controller surfaces
- front-door repo/docs pages agree on current project state
- invariant tests cover the control edges Phase 7 will build on

## Handoff to Phase 7

Only after this stabilization pass should Phase 7 add:

- controller-driven advancement until boundary
- bounded implementation-loop semantics
- minimum typed runtime/operator event surface
- policy-driven variable control decisions
- governance before `sync`

Phase 7 should build on a cleaner controller path, not try to clean it while adding new liveness behavior.
